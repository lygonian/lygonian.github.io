CRM-Sync-Bibliothek

Bevor fünf Projekte je einzeln mit dem CRM sprechen, lohnt eine grundsätzlichere Frage: Wie sähe die eine Bibliothek aus, auf der alle aufbauen können? Die Antwort ist ein inkrementeller GraphQL-zu-CSV-Sync — das Datenfundament meiner gesamten Automatisierungslandschaft.

RolleKonzeption & Umsetzung, eigenständig
KontextMittelständisches Energie-/PV-Unternehmen
StackPython, GraphQL, pandas, requests
StatusProduktiv, Backbone mehrerer Projekte

Ausgangsproblem

Immer mehr Automatisierungen brauchten aktuelle CRM-Daten: Kontakte, Dokumente, Projekte, Termine. Jedes Projekt einzeln an die API anzubinden hätte dieselbe Logik mehrfach erzeugt — mit mehrfachen Fehlerquellen und mehrfacher Last auf der API. Erschwerend kam hinzu, dass die Herstellerdokumentation der CRM-Schnittstelle unvollständig war.

Denklogik

  1. Die API erschließen. Was die Dokumentation nicht hergab, habe ich systematisch selbst ermittelt: Abfragen formulieren, Antworten beobachten, Verhalten dokumentieren — bis ein belastbares Bild der Schnittstelle stand.
  2. Einmal bauen statt fünfmal. Eine Bibliothek mit einheitlicher Sync-Logik für alle vier Ressourcentypen, die jedes Folgeprojekt einfach importiert.
  3. Inkrementell statt Vollabzug. Ein Watermark auf dem Änderungszeitstempel merkt sich den letzten Stand. Eine leichte Abfrage erkennt, was sich geändert hat; nur für diese Datensätze werden die vollen Daten geholt.
  4. Idempotent schreiben. Geänderte Datensätze werden per ID in die lokalen CSV-Bestände eingefügt oder aktualisiert (Upsert). Ein Lauf darf jederzeit wiederholt werden, ohne Duplikate zu erzeugen.
  5. Konsistenz absichern. Der Sync wiederholt sich, bis ein Durchlauf keine Änderungen mehr findet (loop-until-stable) — Änderungen während des Laufs gehen nicht verloren. Bei Überlast oder Serverfehlern greift ein Retry mit exponentiellem Backoff.

Technische Umsetzung

Die Bibliothek ist in Python geschrieben und spricht die CRM-Schnittstelle per GraphQL an. Die zweiphasige Abfrage (leichte Delta-Erkennung, dann volle Daten nur für Geändertes) hält die Last klein. Ergebnisse landen als CSV-Bestände, die nachgelagerte Projekte — vom Vertragsgenerator bis zum Excel-Sync — direkt weiterverwenden. Die Konfiguration läuft über Umgebungsvariablen, Zugangsdaten bleiben außerhalb des Codes.

Ergebnis

Statt fünf Insellösungen gibt es eine gemeinsame Datenschicht: Die Bibliothek läuft produktiv und wird von mehreren internen Projekten als Backbone importiert. Neue Automatisierungen starten nicht mehr bei der API-Anbindung, sondern direkt bei ihrer eigentlichen Aufgabe.

  • VersorgtPV-Vertragsgenerator, Kontakt- und Excel-Syncs
